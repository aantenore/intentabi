# Codex Integration Boundary

IntentABI must not pretend that a Codex skill or hook can transparently replace
the text already submitted from the composer.

The supported hook surface can inspect a user prompt, block it, or add context.
It is useful for policy and telemetry, but it is not treated as an in-place
prompt-rewrite API. Likewise, post-generation processing cannot recover tokens
already generated by a model.

Consequently, token-saving integration has two explicit shapes:

1. **Codex SDK wrapper** — the implemented alpha host owns the request before
   `thread.run`, observes SemWitness preparation, and submits the exact original
   input; a later contract may submit a verified candidate only after promotion;
2. **App Server client** — a UI or service owns `turn/start`, records original
   and transformed digests, and keeps approval/event semantics visible.

Official references:

- [Codex hooks](https://learn.chatgpt.com/docs/hooks)
- [Codex SDK](https://learn.chatgpt.com/docs/codex-sdk)
- [Codex App Server](https://learn.chatgpt.com/docs/app-server)

## Implemented SDK Shadow Contract

`@intentabi/codex-host` composes the public SemWitness `TextRequestPreparer`
port with a provider-neutral `CodexTurnTransport`. The host:

- snapshots trusted bindings and the original string;
- starts preparation without giving the candidate to the transport;
- calls the exact transport once with the original input and unchanged options;
- reduces original, candidate, codec, reason set, and promotion binding to
  scope-bound keyed digests;
- records proof presence without duplicating SemWitness proof semantics, and
  never reflects on opaque turn options or SDK output;
- emits a frozen authenticated evidence envelope with
  `submitted: "original"`;
- provides strict detached verification plus an atomic replay-guard contract;
- returns the same authenticated event on uncertain sink acknowledgement so a
  caller can retry with the original idempotency key;
- treats preparation timeout/fault/malformed output and sink failure as shadow
  evidence only.

Consumers derive `expectedBindingDigest` from their trusted execution and
preparation configuration with `deriveCodexShadowBindingDigest` before reading
an envelope. Copying the digest from an incoming envelope would make the scope
check tautological and is not a supported verification pattern.

`@intentabi/adapter-codex-sdk` creates the thread from one validated frozen
`ThreadOptions` snapshot, then derives the evidence binding from that same
snapshot. Its compile-time contract is pinned to official SDK `0.144.4`.
Native CLI optionals are deliberately ignored in this source-only workspace;
the future runnable composition package must install its platform binary.

The SDK already reports `RunResult.usage` and exposes basic streamed lifecycle
events. `normalizeCodexUsage` maps a caller-supplied SDK-shaped value without
inventing totals or savings; it does not yet claim provider provenance. The
adapter does not collect or persist a binding-correlated receipt, so no measured
token-saving claim exists.

This is functional passthrough instrumentation, not token reduction. The
current SemWitness preparer returns identity unless an exact codec/policy,
tokenizer, deployment scope, and held-out promotion manifest authorize a
verified preparation; user prose is not promoted by this repository.

## Promotion Path

The wrapper cannot enable semantic reuse or transformed submission on its own.
Any future active transformation must declare tokenizer/model/prompt-prefix
scope, preserve tool and approval contracts, measure cold/warm behavior, and
pass SemWitness held-out task-quality, unsafe-accept, and net-value gates.

The next runnable slice composes SDK, SemWitness, host, and sink and records
provider-reported usage for counterbalanced baseline/candidate evaluation. App
Server over local stdio JSONL remains the richer control-plane option for
approval, tool, history, and event parity. WebSocket transport and active
prompt rewriting remain out of scope.

Agentic SDLC remains an application consumer: its plugin may launch the wrapper
or observatory, but the provider-agnostic runtime remains in this separate
repository. SemWitness remains the proof/evaluation authority.
